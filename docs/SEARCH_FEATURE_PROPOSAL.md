# VapeStockHub 搜索功能可开发方案

更新时间：2026-05-21

## 1. 文档目的
- 本文用于把搜索功能从“方向讨论”收敛为“可开发方案”。
- 本文默认遵循 `docs/CURRENT_STAGE.md` 中的阶段定义与归类规则。
- 当前口径下，搜索功能归类为：`MVP+ 优化`。

## 2. 一句话结论
- 当前建议做搜索，但只做 `轻量、高意图、围绕库存与后台工作台` 的搜索。
- 不建议把搜索做成全站统一入口，也不建议把它扩成新的 SEO 体系。

## 3. 为什么现在可以做
- 项目已经上线，`MVP` 主体完成，当前进入收尾与早期优化阶段。
- 搜索不会改变现有商业模型，仍然服务于主链路：
  - `库存展示 -> 私域联系 -> 有效询盘`
- 搜索的价值主要体现在两类场景：
  - 买家更快找到目标库存
  - 内部更快定位待处理提报和库存记录

## 4. 目标定义

### 4.1 对公开站的目标
- 让高意图买家能够更快从列表页进入目标详情页。
- 降低“我知道品牌/型号/风味/市场，但列表翻找太慢”的摩擦。
- 提升 `Inventory -> Detail -> Contact` 路径效率。

### 4.2 对后台的目标
- 让运营快速找到：
  - 待审核提报
  - 待发布草稿
  - 指定品牌/市场/仓库/型号记录
- 降低翻页和人工扫描成本。

## 5. 当前推荐范围

### 5.1 `P0` 公开站：`/inventory` 搜索
- 页面：`/inventory`
- 定位：库存列表页内的关键词搜索，不单独新建复杂搜索页
- 搜索对象：
  - `brand`
  - `title`
  - `product_type`
  - `flavor`
  - `market`
  - `featured_markets`
  - `warehouse_location`
- 交互方式：
  - 输入关键词
  - 提交后进入 `/inventory?q=keyword`
  - 与现有筛选共存：`brand / market / sort / page`

### 5.2 `P0` 后台：`/admin/submissions` 队列搜索
- 页面：`/admin/submissions`
- 搜索对象：
  - `supplier_name`
  - `brand`
  - `model_name`
  - `target_market`
  - `warehouse_location`
- 目标：快速定位某一条提报或某类待处理记录
- 交互方式：
  - 与当前状态筛选并存
  - 搜索后仍停留在当前队列语境

### 5.3 `P0` 后台：`/admin` 库存区搜索
- 页面：`/admin`
- 搜索对象：
  - `title`
  - `slug`
  - `brand`
  - `market`
  - `warehouse_location`
- 目标：快速找到待发布草稿、某品牌库存、某市场库存
- 交互方式：
  - 与当前 `status` 筛选并存
  - 搜索后仍可继续进行发布或编辑动作

## 6. 当前明确不做
- 不做全站统一搜索
- 不做 Header 常驻大搜索框
- 不做博客、政策页、库存页混合搜索
- 不做联想建议、拼写纠错、复杂相关度算法
- 不做 Algolia / Elasticsearch / Meilisearch 接入
- 不做可索引的独立搜索结果页
- 不做把 `Brand / Market / Price` 聚合页改造成搜索页

## 7. 页面位置建议

### 7.1 `/inventory`
- 放在页面主标题说明区下方、结果统计与排序区上方。
- 推荐结构：
  - H1 与说明文案
  - 搜索框
  - 当前筛选 / 当前搜索词
  - 结果总数与排序
  - 卡片列表
- 原因：
  - 这里是当前最自然的找货页
  - 用户进入这里时意图最强
  - 不会破坏首页的品牌叙事和 SEO 导航结构

### 7.2 `/admin/submissions`
- 放在“队列记录”标题区右侧，和状态切换处于同一区域。
- 原因：
  - 用户先看队列上下文，再输入关键词筛定位目标
  - 不会和卡片优先级信息冲突

### 7.3 `/admin`
- 放在“待发布草稿 / 当前状态库存”模块标题区域。
- 原因：
  - 用户当前本来就在这个区域进行状态管理
  - 搜索放在这里最符合工作台动作路径

### 7.4 Header
- 当前只建议保留“跳转到 `/inventory`”的轻入口。
- 不建议首版直接在 Header 放完整搜索输入框。

## 8. 公开站功能边界

### 8.1 必备功能
- 关键词搜索
- 与当前筛选并存
- URL 参数可分享
- 清空搜索
- 结果数反馈
- 空结果提示
- 搜索后分页保持正确
- 参数页 `noindex`

### 8.2 搜索逻辑建议
- 默认做“包含匹配”而不是复杂权重排序。
- 若命中多个字段，不做复杂分值体系，先统一返回。
- 排序逻辑继续沿用当前规则：
  - 默认最新
  - 可切换价格升序
- 先保证“可用”和“稳定”，再谈相关度。

### 8.3 空结果处理
- 空结果时不只显示 `No results`。
- 应同时给出：
  - 清空搜索
  - 返回全部库存
  - 推荐继续用品牌 / 市场筛选
- 目标是避免买家在第一次搜索失败后直接流失。

## 9. 后台功能边界

### 9.1 `/admin/submissions`
- 搜索必须与状态筛选同时生效。
- 搜索后分页仍然可用。
- 搜索命中后，卡片结构不变，不新增复杂表格。
- 目标是“更快定位”，不是重做审核页信息架构。

### 9.2 `/admin`
- 搜索必须与库存状态筛选同时生效。
- 搜索结果仍保留当前动作入口：
  - 去发布检查
  - 去编辑库存
  - 直接发布
- 目标是“提效”，不是重做后台总控台。

## 10. SEO 规则
- 搜索参数页不应参与索引。
- `/inventory?q=...` 及其组合参数页统一保持：
  - `canonical -> /inventory`
  - `robots -> noindex, follow`
- 原因：
  - 搜索结果页天然容易产生大量低质量参数页
  - 当前 SEO 主阵地仍应是：
    - Inventory
    - Detail
    - Brand
    - Market
    - Price
- 搜索功能的目标是提升站内找货效率，而不是承接新的搜索引擎收录结构。

## 11. 技术实现建议

### 11.1 首版建议
- 基于当前 `Supabase + Server Components` 实现。
- 使用 URL 参数驱动：
  - `/inventory?q=...`
  - `/admin?inventory_status=...&q=...`
  - `/admin/submissions?status=...&q=...`
- 公开站首版建议服务端查询 + 本地过滤组合实现。
- 后台首版建议直接在服务端查询层叠加 `ilike` 条件。

### 11.2 为什么不需要重型搜索方案
- 当前数据规模仍然较小。
- 搜索字段结构明确。
- 当前最重要的是：
  - 快速落地
  - 成本可控
  - 维护简单
- 在未验证搜索使用频率前，不应提前引入新的基础设施复杂度。

### 11.3 未来升级触发条件
- 当活跃库存显著增长
- 当搜索慢到影响使用
- 当命中质量明显下降
- 当后台运营对搜索依赖度明显提高
- 满足以上条件后，再考虑 PostgreSQL Full Text Search，而不是外部搜索引擎优先

## 12. 埋点与观察建议
- 建议至少记录以下事件：
  - `search_performed`
  - `search_term`
  - `search_result_count`
  - `search_page_type`
  - `search_clicked_listing`
- 页面维度建议区分：
  - `inventory`
  - `admin_inventory`
  - `admin_submissions`
- 目标不是做完整 BI，而是验证：
  - 搜索有没有被真实使用
  - 搜索后是否更容易进入详情页或编辑页

## 13. 实施优先级

### 13.1 第一优先级
- `/inventory` 搜索
- `/admin/submissions` 搜索
- `/admin` 搜索

### 13.2 第二优先级
- Header 增加轻量跳转入口
- 搜索状态展示优化
- 空结果推荐优化

### 13.3 当前不进入排期
- 聚合页内搜索
- 搜索联想
- 高级筛选组合面板
- 搜索行为推荐

## 14. 建议开发顺序
1. 先做 `/inventory` 搜索
2. 再做 `/admin/submissions` 搜索
3. 再做 `/admin` 搜索
4. 最后统一补埋点与细节体验

## 15. 验收标准

### 15.1 `/inventory`
- 用户可通过关键词缩小库存列表
- 搜索与筛选、排序、分页不冲突
- 搜索参数页不被索引
- 空结果页有明确下一步动作

### 15.2 `/admin/submissions`
- 运营可通过关键词快速定位目标提报
- 搜索与状态筛选可同时生效
- 搜索不会破坏当前审核动作路径

### 15.3 `/admin`
- 运营可快速定位目标库存或草稿
- 搜索与状态筛选可同时生效
- 搜索后仍可继续原有编辑/发布动作

## 16. 当前结论
- 搜索功能适合现在做，但必须做轻。
- 当前最合理的定义是：
  - `搜索 = MVP+ 优化`
  - `范围 = 库存找货 + 后台提效`
  - `首版 = 不做复杂搜索系统`
- 若后续要扩成更大范围，应重新按 `第二阶段扩张` 规则单独评估。
