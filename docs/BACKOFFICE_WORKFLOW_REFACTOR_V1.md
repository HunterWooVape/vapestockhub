# VapeStockHub 后台工作流重构方案 V1

## 1. 文档定位

本文件是当前后台重构的主基线文档。

目标不是继续零散修页面，而是一次性统一以下 5 件事：

- 内部人员到底按什么链路工作
- 每个后台页面到底负责什么
- 哪些整理交给规则层，哪些交给真实 LLM
- 前后台字段如何拆层，避免原始值直接污染发布值
- 现有历史文档如何继续保留、更新或降级为参考

本轮确认后的统一心智为：

`打开 submit-stock -> 录入 -> LLM 完善 -> 审核 -> 发布`

---

## 2. 本轮已确认决策

### 2.1 工作流主链路

内部主链路正式统一为：

1. 打开 `submit-stock`
2. 完成最小录入
3. 进入 `LLM 完善`
4. 进入 `审核`
5. 进入 `发布`

底层仍允许存在 `draft`，但不应让操作人员感受到“多一层系统复杂度”。

### 2.2 `submit-stock` 入口策略

`submit-stock` 不再继续作为“半公开 + access code”入口。

正式改为：

- 仅后台登录后可进入
- 复用现有 `Admin / Staff` 登录态
- 取消独立 `code` 验证

原因：

- 当前已不再需要对外共用入口
- access code 与后台登录双轨并存，容易制造链路断裂和假成功体感
- 内部流程应追求连续，而不是增加额外校验步骤

### 2.3 `market` 字段策略

`market` 主字段新增：

- `Global`

同时新增一个用于补充限制条件的说明字段，建议命名：

- `market_access_note`

或

- `restricted_markets_note`

推荐优先使用：

- `market_access_note`

字段分工如下：

- `market`：用于前台聚合、SEO、筛选的主分类字段
- `market_access_note`：用于记录 `Except UAE`、`Not for Saudi Arabia` 等销售限制说明

不建议把复杂限制规则直接塞进 `market` 主字段。

### 2.4 结构化单位策略

以下字段优先由规则层标准化，而不是等待 LLM：

- `puff`
- `nicotine`
- `e_liquid`

执行原则：

- 后台录入允许输入原始数值，如 `20000`、`5`、`21`
- 规则层负责补齐展示单位
- 前台统一标准化展示

推荐展示口径：

- `puff` -> `20,000 Puffs`
- `nicotine` -> `5%`
- `e_liquid` -> `21 mL`

### 2.5 真实 LLM 的落点策略

真实 LLM 进入实施计划，但不做全站泛化接入。

第一落点统一定为：

- `/admin/submissions/[id]`

第一阶段只做建议，不直接覆盖最终发布值。

本轮先完成文档埋点与字段/页面预留，不立即启动真实 LLM 接入开发。

触发时机约定为：

- 冷启动数据进入稳定录入期后
- 提交与审核主链路已跑顺
- 规则层标准化与发布阻断已先落地

届时应直接沿用本文件既定边界，不重新发散讨论：

- 只先接 `/admin/submissions/[id]`
- 只输出结构化建议
- 只做人机协同，不自动发布

### 2.6 前台内容拆分策略

当前详情页中单一的 `Inventory Manifest & Details` 需要拆分为 3 个层次：

1. `Product Overview`
   - 用于英文产品与库存概览
   - 来自规则清洗 + LLM 总结

2. `Inventory Manifest`
   - 只展示口味明细、数量拆分、批次清单等 manifest 内容
   - 保持偏原始、事实型

3. `Commercial Notes`
   - 汇总 `库存备注 / 包装备注 / 补充备注`
   - 使用英文整理后的结果

### 2.7 原始中文输入策略

`submit-stock` 允许内部输入中文、脏值、非标准写法。

但发布层必须满足：

- 前台展示字段使用英文
- 关键标准字段符合 canonical value
- 中文原始备注不得直接裸展示到前台

## 3. 当前问题总诊断

本轮测试暴露出的主要问题，不是单个 bug，而是 4 类结构性问题叠加：

### 3.1 流程心智不统一

- 文档与页面都在说“流程已简化”
- 但实际页面仍保留多入口、多解释、多跳转
- 操作人员很难一眼知道“我现在在哪一步，下一步去哪”

### 3.2 数据分层不清

当前很多字段还没有严格区分：

1. 原始输入值
2. 规则标准化值
3. LLM 建议值
4. 最终发布值

结果就是：

- 录入值直接流到前台
- 中文备注直接出现在前台
- 大小写、单位、写法不统一
- 说明类内容全部堆进单一 `description`

### 3.3 页面职责过载

当前多个页面同时在做两种事：

- 执行任务
- 教用户如何执行任务

这会导致：

- 页面解释很多
- 视觉主次不清
- 用户找不到最强主动作

### 3.4 “智能整理”能力被高估

当前系统存在的主要是轻量规则清洗，不等于真正的 LLM 整理能力。

因此目前它能处理：

- 去空格
- 补部分格式
- 做少量合并

但还不能稳定处理：

- 中文备注转英文可发布内容
- 英文产品介绍生成
- manifest 与 notes 分层
- 较复杂的 canonical mapping

## 4. 目标工作流

## 4.1 操作人员视角

内部人员只需要记住这一条：

`打开 submit-stock -> 录入 -> LLM 完善 -> 审核 -> 发布`

解释：

- `录入`：把原始资料送进系统
- `LLM 完善`：把原始资料整理成更像库存候选的结果
- `审核`：人工确认缺失项、标准化和风险
- `发布`：最终把内容变成前台可展示 inventory

## 4.2 系统承接视角

系统内部仍按以下页面承接：

1. `/submit-stock`
2. `/admin/submissions`
3. `/admin/submissions/[id]`
4. `/admin/edit/[id]`
5. `status = active`

系统内部真实链路统一理解为：

`原始录入 -> 提报队列 -> LLM 建议 -> 审核确认 -> 草稿修正 -> 发布`

---

## 5. 页面职责重构

## 5.1 `/submit-stock` -> `库存录入台`

### 页面目标

- 用最短时间录入一条 submission
- 不做复杂讲解
- 不承担发布职责

### 页面主动作

- `提交并进入 LLM 完善`

### 页面应保留

- 最低必填字段
- 结构化录入分组
- 本地草稿恢复
- 极短字段提示

### 页面应删除或明显弱化

- access code 验证
- 大段教学说明
- 与当前提交动作无关的摘要面板
- “提交后流程”型长说明区

### 页面完成后应发生什么

- 提交成功后不要只停留在当前页显示提示
- 应直接把人送到后续处理入口
- 理想状态是直达该 submission 的处理页，次优为直达提报队列

## 5.2 `/admin/submissions` -> `提报队列`

### 页面目标

- 让人快速判断先处理哪条
- 一眼看出下一步动作
- 不再承担复杂说明和多重入口展示

### 页面主动作

- `进入处理`

### 每条记录最重要的信息

- 品牌 / 型号
- 当前状态
- 缺失项数量
- 是否已有 LLM 结果
- 是否已转草稿
- 下一步动作

### 应弱化

- 重复性的解释文字
- 多个意思相近的按钮
- 需要用户自己猜区别的动作名称

### 目标心智

这页不是“看记录”，而是“排队推进下一步”。

## 5.3 `/admin/submissions/[id]` -> `LLM 完善与审核`

### 页面目标

- 统一承接 LLM 建议与人工审核
- 清楚表达“先补齐，再整理，再确认”

### 页面主动作顺序

1. 补齐最低必填
2. 生成 / 刷新 LLM 建议
3. 审核标准化结果
4. 转草稿

### 页面第一屏必须回答的 3 个问题

- 这条 submission 缺什么
- LLM 是否已生成建议
- 我做完这里后下一步去哪

### 应删除或弱化

- 与当前任务无关的大段说明
- 重复的原始信息展示
- 多个命名相近但职责重叠的按钮

## 5.4 `/admin/edit/[id]` -> `发布前确认`

### 页面目标

- 明确这是 `AI 之后、发布之前` 的最终整理页
- 聚焦“能不能发布”

### 页面主动作

- `保存修改`

### 次级关键动作

- `切换状态`

### 页面必须表达清楚

- 当前草稿来自哪条 submission
- 当前内容是否已过发布门槛
- 哪些项仍阻塞发布

## 5.5 `/admin` -> `后台总控台`

### 页面目标

- 只做总入口和摘要
- 不再承载日常主作业表单

### 对 `Staff`

- 默认强调进入 `提报队列`
- 不继续把大表单和高级入口放在主视觉中心

### 对 `Admin`

- 保留总控与高级入口
- 但低频动作默认折叠或下沉

## 6. 数据分层重构

## 6.1 四层数据心智

后台字段统一按四层理解：

1. `原始输入值`
2. `规则标准化值`
3. `LLM 建议值`
4. `最终发布值`

### 含义

- `原始输入值`：允许脏、允许中文、允许省事写法
- `规则标准化值`：通过代码可稳定修正的格式和 canonical value
- `LLM 建议值`：通过模型生成的英文整理和结构建议
- `最终发布值`：人工确认后真正进入前台的值

## 6.2 哪些必须优先放在规则层

以下内容不应等待 LLM：

- 数字格式与千分位
- 单位补齐
- 大小写与空格归一
- `brand` 的已知 canonical mapping
- `market` 的已知 canonical mapping
- `product_type` 的固定归一

## 6.3 哪些适合交给 LLM

以下内容适合第一阶段真实 LLM：

- 英文 `Product Overview`
- 英文 `Commercial Notes`
- `Inventory Manifest` 的结构整理
- 原始备注中的风险与缺失提炼
- 不确定标准值的候选建议

## 6.4 哪些不交给 LLM

- 自动发布
- 自动确认价格真实性
- 自动决定是否合规可卖
- 自动替代人工修改最终发布值

## 7. 字段与前台内容结构

## 7.1 当前需要新增或明确职责的字段

建议引入或明确以下字段职责：

- `market`
- `market_access_note`
- `flavor_breakdown`
- `stock_notes`
- `packaging_notes`
- `extra_notes`
- `overview_text` 或 `description_summary`
- `commercial_notes`

若短期不立即扩表，也要先在逻辑上明确映射关系。

## 7.2 前台详情页目标结构

### A. `Product Overview`

来源：

- 结构字段 + LLM 总结

特点：

- 英文
- 面向买家可读
- 不等于原始备注拼接

### B. `Inventory Manifest`

来源：

- `flavor_breakdown`
- 其他批次或清单型明细

特点：

- 保持清单感
- 事实信息优先
- 不做营销改写

### C. `Commercial Notes`

来源：

- `stock_notes`
- `packaging_notes`
- `extra_notes`

特点：

- 英文
- 汇总后的交易备注
- 用于买家理解交付、包装、补充条件

## 8. 真实 LLM 接入策略

## 8.1 为什么现在要提上日程

你这轮测试已经说明：

- 规则层足以解决单位、大小写、基础格式
- 但不足以解决英文整理、中文备注转发布内容、前台内容分层

所以真实 LLM 已经从“可选优化项”变成“必要生产力项”。

## 8.2 为什么只先接一个落点

如果现在全链路同时接 LLM，会同时引入：

- 输出不稳定
- 调试困难
- 页面职责再次模糊
- 返工成本高

因此第一阶段只接：

- `/admin/submissions/[id]`

## 8.3 第一阶段 LLM 输出结构

建议统一输出结构化 JSON，而不是自由文本直写：

- `normalizedFields`
- `overviewCandidate`
- `manifestCandidate`
- `commercialNotesCandidate`
- `missingFields`
- `riskFlags`
- `humanReviewFocus`

## 8.4 第一阶段验收标准

- 人工能看懂 LLM 输出在帮什么
- 人工可以选择性采用，而不是被动接受覆盖
- 输出能明显缩短 submission review 到 draft 的时间

## 9. 登录、角色与入口策略

## 9.1 登录策略

正式统一为：

- 后台只保留一套登录体系
- `submit-stock` 与 `/admin*` 共用同一后台会话

## 9.2 角色策略

继续只保留：

- `Admin`
- `Staff`

### `Staff`

负责：

- 录入
- LLM 完善
- 审核
- 转草稿
- 草稿修正

### `Admin`

负责：

- 最终发布判断
- 高风险动作
- 异常处理

## 9.3 入口策略

### `Staff` 默认主入口

- `/submit-stock`
- `/admin/submissions`

### `Admin` 默认主入口

- `/admin`
- `/admin/submissions`

---

## 10. 本轮明确不做的事

本轮重构严格不做：

- 复杂权限矩阵
- 复杂审批流
- 对外供应商自助后台
- 多团队后台
- LLM 自动发布
- 把后台做成 ERP

---

## 11. 重构完成后的验收口径

## 11.1 流程验收

内部人员在不看说明文档的前提下，应能完成：

- 登录
- 录入 submission
- 找到下一步处理页
- 生成 LLM 建议
- 转草稿
- 进入发布前确认页

## 11.2 页面验收

每个核心页面都必须满足：

- 只有一个最强主动作
- 首屏不靠长文解释工作流
- 能回答“我现在在哪一步”
- 能回答“我下一步去哪”

## 11.3 内容验收

前台最终展示内容必须满足：

- 英文
- 单位完整
- manifest 与 notes 分层清楚
- 不直接暴露中文原始备注

---

## 12. 这份文档与其他文档的关系

从本轮开始，后台相关文档按以下关系理解：

- 本文件：后台重构主基线
- `BACKOFFICE_REFACTOR_EXECUTION_PLAN.md`：按阶段落地顺序
- `DOCS_BACKOFFICE_STATUS_MATRIX.md`：文档保留、更新、归档判断
- 既有 workflow / IA / checklist 文档：继续保留，但需按本文件重新对齐

## 13. 结论

这次后台问题的核心，不是“某个页面写得不够好”，而是后台仍然带着历史过渡态。

本轮统一结论是：

- 用一条主链路收口
- 用一套登录收口
- 用规则层解决结构问题
- 用真实 LLM 解决英文整理和内容分层问题
- 用页面职责收口认知负担

后续所有后台实现，应以本文件为准逐阶段落地，避免继续边做边跳。
