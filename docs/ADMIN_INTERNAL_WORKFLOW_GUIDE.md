## 1. 文档目的

本文件用于把当前后台工作流重新收口成一条更短、更好执行的链路。

目标不是描述所有技术细节，而是让内部操作人员每天都清楚：

- 先去哪一页
- 先做什么
- LLM 在哪里介入
- 最后在哪里发布

---

## 2. 当前正式工作流

### 2.1 操作心智

当前后台统一操作心智为：

`submit-stock -> 提报队列 -> 提报审核 -> 发布前确认 -> 发布`

这是给操作人员的前台心智，不要求大家记住所有中间技术节点。

### 2.2 技术承接链路

系统内部继续由以下页面承接：

1. `submit-stock`
2. `admin/submissions`
3. `admin/submissions/[id]`
4. `admin/edit/[id]`
5. 发布为 `active`

对技术实现的理解应统一为：

`录入 -> 提报队列 -> 审核与转草稿 -> 最终修正 -> 发布`

### 2.3 为什么保留 `draft`

虽然运营心智已经极简，但技术上仍然保留 `draft` 层，原因是：

- 便于补图
- 便于修正 title / slug / description
- 便于保留发布前检查
- 便于隔离误发布风险

## 3. 页面职责

### 3.1 `/submit-stock`

当前定位：

- 内部录入台
- 原始资料结构化入口

适合做的事：

- 快速录入一条库存的基础字段
- 粘贴原始 notes、flavor、image links
- 把 Excel 整理后的内容送进系统

不适合做的事：

- 直接发布
- 写最终展示文案
- 替代后续复核

### 3.2 `/admin/submissions`

当前定位：

- 录入后队列页
- 每天处理新 submission 的起点

适合做的事：

- 查看新录入记录
- 识别哪些记录需要补字段
- 进入单条审核页

### 3.3 `/admin/submissions/[id]`

当前定位：

- LLM 介入前后的审核中间站

这里负责：

- 补齐最低必填字段
- 查看原始快照
- 触发 LLM 整理结果
- 确认是否进入 `draft`

操作心智上，这里就是：

- `LLM 完善`
- `快速复核`

### 3.4 `/admin/edit/[id]`

当前定位：

- 发布前最终整理页

这里负责：

- 修正正式标题
- 确认 slug
- 确认价格、数量、市场、仓库
- 补图
- 检查前台可展示性
- 决定是否发布

---

## 4. 标准日常操作

### 4.1 新库存的标准路径

推荐按以下顺序处理：

1. 从供应商处拿到 Excel 或原始资料
2. 内部先做一轮人工整理
3. 打开 `/submit-stock` 完成录入
4. 去 `/admin/submissions` 找到这条记录
5. 进入 `/admin/submissions/[id]`
6. 先补齐最低必填字段，再决定是否生成 AI 建议
7. 满足条件后转草稿
8. 进入 `/admin/edit/[id]`
9. 补图、修正文案、确认字段
10. 由 `Admin` 决定是否发布

### 4.2 最短执行口诀

每天按这句执行即可：

`先录入，后 AI；先复核，后发布。`

---

## 5. LLM 的使用规则

### 5.1 什么时候使用

当最低必填字段已齐全后，再触发 LLM 动作。

最低必填字段未齐时：

- 不继续 LLM
- 先补字段

### 5.2 LLM 负责什么

LLM 当前负责：

- 字段标准化
- 结构整理
- 草稿内容辅助生成
- 风险和缺失提示

### 5.3 LLM 不负责什么

LLM 当前不负责：

- 自动发布
- 最终真实性判断
- 价格审批
- 替代人工复核

---

## 6. 审核与发布规则

### 6.1 人工必须确认的项目

正式发布前，人工必须确认：

- `price`
- `quantity`
- `market`
- `warehouse_location`
- 图片可用性
- 描述可读性
- 是否真的值得上线

### 6.2 不应直接发布的情况

以下情况不应直接发布：

- 关键字段缺失
- 图片无效
- 标题或描述明显不清
- 市场或仓库不可信
- 价格异常未复核

---

## 7. 角色权限

### 7.1 当前是否保留

当前仍然建议保留，但只保留最小边界。

正式建议：

- `Staff` 负责录入、整理、LLM、修 draft
- `Admin` 负责最终发布、删除、异常处理

### 7.2 为什么仍要保留

即使流程极简，发布动作依然高风险，因为它会直接影响：

- 前台页面质量
- SEO 质量
- 买家第一印象
- 询盘质量

因此不建议把所有权限完全打平。

### 7.3 冷启动阶段如何理解

如果当前只有一人操作，现实执行上可以是同一个人。

但系统层仍建议保留最小权限边界，避免未来多人协作时返工。

---

## 8. 异常处理

### 8.1 `submit-stock` 提交失败

建议动作：

- 按高亮字段回去补齐
- 使用保留的本地草稿继续填写

### 8.2 LLM 动作不可用

常见原因：

- 最低必填字段未齐

建议动作：

- 先补齐缺失字段
- 再执行 LLM 动作

### 8.3 进入 `draft` 后仍不够好

这是正常情况。

建议动作：

- 在 `/admin/edit/[id]` 继续修正
- 先保证真实和清晰
- 再考虑发布

---

## 9. Workflow Review 检查清单

后续检查整条链路时，重点看以下项目：

- `submit-stock` 是否能稳定录入
- 缺失字段是否会阻止提交
- `/admin/submissions` 是否能快速找到新记录
- `/admin/submissions/[id]` 是否能顺利触发 LLM
- LLM 输出后是否能顺利进入 `draft`
- `/admin/edit/[id]` 是否能顺利补图和修字段
- `Staff` 是否不能直接发布
- `Admin` 是否仍可完成最终发布

---

## 10. 当前结论

当前后台不再强调“多步骤流程”，而强调“少步骤完成任务”。

正式结论如下：

- `submit-stock` 是内部录入台
- `提报审核` 是补齐、AI 建议、转草稿的中间站
- `发布前确认` 是发布前最后一道整理页
- `Admin / Staff` 只保留最小边界
- 后续优化重点应放在减少点击和减少重复动作，而不是继续扩流程
