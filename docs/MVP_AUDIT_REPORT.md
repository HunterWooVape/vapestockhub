## 7. 当前遗留问题与未收尾事项

### 7.1 P0：正式收口前必须完成
1. 完成一轮完整正式环境 walkthrough：
   - `首页 -> 列表 -> 详情 -> Telegram/WhatsApp`
   - `后台登录 -> submit-stock -> submissions -> edit -> publish`
   - `leads` 是否正常回流
2. 接通 `MONITORING_WEBHOOK_URL`，避免线上异常只能靠手看日志。
3. 确认生产数据库已备份或已有恢复点。
4. 确认真实运营所需环境变量已完全对齐。

### 7.2 P1：短期内应尽快补齐
1. 增加最小 CI 流水线。
2. 增加最小自动化 smoke test。
3. 补齐 README 与实际环境变量之间的文档差异：
   - `NEXT_PUBLIC_GA_MEASUREMENT_ID`
   - `GOOGLE_SITE_VERIFICATION`
4. 完成 Search Console 验证接入。
5. 确认 GA4 数据是否真实回流，而不是仅代码预留。

### 7.3 P1：文档一致性问题
1. 第二阶段文档已修正 `/submit-stock` 采用“后台身份校验”。
2. 当前实际代码逻辑是：`/submit-stock` 依赖后台登录态，而不是 access code。
3. 这意味着：
   - 当前实现更偏内部工作台
   - 文档描述已与实现存在偏差
4. 需要后续统一文档基线，避免再次多版本并行。

### 7.4 P2：质量与运营优化项
1. 后台 walkthrough 文档体系仍有待统一更新。
2. 自动化测试几乎为空，当前主要依赖人工验证。
3. 告警链路未接通，会放大线上风险发现时延。
4. 真实数据覆盖已经存在，但首页和列表中的部分业务字段表现仍有继续优化空间。

## 8. 当前发现的潜在问题

### 8.1 监控侧问题
- 线上 `/health` 显示 `webhookEnabled = false`。
- 说明监控日志虽存在，但没有主动告警提醒。
- 一旦线上异常发生，发现速度可能过慢。

### 8.2 自动化保障不足
- 当前没有成体系的 `test` 脚本。
- 这意味着每次变更后，核心链路仍主要依赖人工回归。

### 8.3 文档与实现轻微漂移
- `submit-stock` 的安全模型在文档中仍有旧描述。
- 环境变量文档与真实代码依赖字段未完全对齐。

### 8.4 运营数据真实性风险
- 线上已有真实库存承接，但后续若没有稳定的复核节奏、看板和告警，库存新鲜度会逐步失真。

## 10. 短期推进计划（仅保留 1-2 个月）

说明：

- 本节负责给出短期方向与优先级。
- 若要逐项执行和勾选完成状态，应以 `docs/MVP_CLOSURE_CHECKLIST.md` 为准。

### 10.1 短期目标
- 目标不是继续扩功能。
- 目标是把当前项目从“主体已完成”推进到“可稳定运营、可稳定承接真实询盘、可阶段性结案”。

### 10.2 短期优先级排序

#### P0：上线收口
1. 做正式环境全链路 walkthrough
2. 接通 webhook 告警
3. 确认生产库备份与恢复点
4. 复核环境变量与部署配置
5. 确认 `leads` 数据真实写入与可查

#### P1：质量保障
1. 增加最小 CI
2. 增加最小 smoke test
3. 补齐 README / 部署文档 / 环境变量文档
4. 统一 `submit-stock` 安全模型文档

#### P1：运营闭环
1. 建立首批真实库存复核节奏
2. 建立每周复盘模板
3. 明确有效询盘的口径
4. 明确首页 / Inventory / Market / Brand 的优先维护名单

#### P2：数据与 SEO 微调
1. 检查低库存页的索引口径是否与当前线上库存匹配
2. 观察 Search Console 初始表现
3. 优化高价值详情页和重点 Market / Brand 页
4. 修正首页与列表页中不够稳定的库存展示细节

### 10.3 建议执行顺序

#### 第 1 周
- 正式环境 walkthrough
- webhook 接通
- 数据库备份确认
- 文档对齐

#### 第 2 周
- 最小 CI
- 最小 smoke test
- Search Console 接入
- GA4 数据确认

#### 第 3-4 周
- 建立周复盘节奏
- 核查真实库存复核机制
- 观察高价值页面表现
- 修正重点页面问题

#### 第 5-8 周
- 按询盘与页面表现做一轮小规模优化
- 关闭仍未收口的上线遗留项
- 形成阶段性复盘和结案记录

### 10.4 资源需求预估
- 工程：`1` 名全栈即可推进
- 运营：`0.5` 名数据/提报/复核支持
- SEO/内容：`0.2 - 0.5` 名兼职支持即可

### 10.5 短期风险预判
- 未接告警导致线上异常发现滞后
- 无自动化测试导致改动后回归成本高
- 文档口径漂移导致后续执行偏离
- 真实库存复核不稳定导致前台数据逐步失真

## 11. 何时可以把 MVP 视为阶段性结案

### 11.1 我的判断标准
满足以下条件后，可以把 `MVP` 视为阶段性结案：

1. 正式站点核心链路已完成真实环境 walkthrough
2. 监控告警已接通
3. 最小 CI / smoke test 已补齐
4. 文档与当前实现已重新对齐
5. 真实库存复核节奏已明确并开始执行
6. 至少跑过一轮真实运营观察周期

### 11.2 结论
- 不是“把短期规划写出来”就算 `MVP` 结案。
- 而是“把短期规划里的收尾动作执行完并验收通过”之后，才更适合把 `MVP` 判定为阶段性结案。
- 更准确地说，应以 `docs/MVP_CLOSURE_CHECKLIST.md` 中的核心收口项完成情况作为判断依据。

### 11.3 更务实的说法
- 现在可以说：`MVP 已基本做完`
- 等短期收尾项执行完，可以说：`MVP 阶段正式结案`
