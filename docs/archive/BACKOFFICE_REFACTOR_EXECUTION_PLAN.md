# VapeStockHub 后台工作流重构执行顺序方案

## 0. 当前执行状态（2026-04）

当前后台工作流收口已完成到阶段 4，进入：

- `轻量人工 walkthrough`
- `问题回写与小范围修正`

当前已完成的第一轮收口包括：

- 阶段 0：文档基线统一
- 阶段 1：后台登录、返回链路、`submit-stock` 入口并轨
- 阶段 2：`submit-stock` 内部录入台化
- 阶段 3：`/admin/submissions` 与 `/admin/submissions/[id]` 的队列化、审核化
- 阶段 4：`/admin` 与 `/admin/edit/[id]` 的总控台化、发布前确认化
- 主流程位置条、主动作文案、详情页首屏摘要、右侧重复信息降噪、列表优先级提示，均已完成第一轮统一

当前明确暂不继续外扩：

- 不提前进入阶段 5 的前台内容拆分开发
- 不提前进入阶段 6 的真实 LLM 接入
- 不提前展开阶段 7 的培训型大文档编写

当前最合适的动作不是继续加功能，而是：

1. 先按真实操作走一轮主链路
2. 把 `顺 / 卡 / 断` 问题点记录下来
3. 只修影响业务跑通或明显影响效率的问题
4. walkthrough 稳定后再判断是否进入阶段 5

## 1. 文档目的

本文件用于承接：

- `BACKOFFICE_WORKFLOW_REFACTOR_V1.md`

目标是把“应该怎么改”拆成“应该按什么顺序改”。

本方案的核心原则只有一条：

`先收口流程骨架，再补字段分层，再接 LLM，最后微调前台展示`

这样做的原因是：

- 如果先改前台展示，后续还会因为字段分层返工
- 如果先全量接 LLM，问题会被模型输出掩盖，难以定位
- 如果不先收口登录和页面职责，后面的任何优化都会继续混乱

---

## 2. 总执行原则

## 2.1 先修主链路，再修装饰层

优先保证：

- 登录连续
- 提交后不丢上下文
- 页面知道下一步去哪

而不是优先做：

- 视觉润色
- 大量微文案优化
- 低频高级能力

## 2.2 先规则，后 LLM

以下问题优先用代码规则解决：

- 单位
- 大小写
- 千分位
- 已知 canonical value

以下问题再交给 LLM：

- 英文概述
- 中文备注转英文
- notes 汇总
- 复杂清单整理

## 2.3 先 review 落点，再扩到其他页面

LLM 第一阶段只进入：

- `/admin/submissions/[id]`

不同时扩散到：

- `submit-stock`
- `/admin/edit/[id]`
- 前台渲染层

## 2.4 每阶段都必须可回归验证

每做完一个阶段，都必须至少验证：

- 是否能顺畅进入下一页
- 页面主动作是否清楚
- 文档与真实行为是否一致

---

## 3. 阶段拆解总览

建议按 8 个阶段推进：

1. 阶段 0：文档基线统一
2. 阶段 1：入口与登录链路收口
3. 阶段 2：`submit-stock` 极简化
4. 阶段 3：提报队列与审核页重构
5. 阶段 4：发布前确认页与总控台收口
6. 阶段 5：字段分层与前台内容拆分
7. 阶段 6：真实 LLM 第一落点接入
8. 阶段 7：全链路内部培训与交接文档

执行顺序必须保持，不建议跳步。

---

## 4. 分阶段方案

## 阶段 0：文档基线统一

### 目标

把这轮后台调整的主基线写清楚，避免旧文档继续互相打架。

### 本阶段内容

- 产出 `BACKOFFICE_WORKFLOW_REFACTOR_V1.md`
- 产出本执行顺序文档
- 梳理 `docs` 中文档的保留、更新、参考关系
- 给关键旧文档补充状态说明

### 依赖

- 无

### 完成标准

- 团队后续讨论统一引用新文档
- 不再同时把多个旧文档都当成主基线

### 风险

- 如果不先做这一步，后面会继续出现“文档说 A，页面做 B，执行按 C”的问题

---

## 阶段 1：入口与登录链路收口

### 目标

彻底收口后台入口，消除 `submit-stock` 与后台主链路之间的割裂。

### 本阶段内容

- 取消 `submit-stock` 的 access code 模式
- 改为后台登录后访问
- 未登录访问时，登录后正确返回目标页
- 提交 submission 后不再仅停留当前页提示，而是直接引导到下一步处理页面
- 补上 submit action 的错误处理，避免假成功

### 涉及页面

- `/submit-stock`
- `/admin/login` 或现有后台登录入口
- 提交 action / server action

### 依赖

- 阶段 0 完成

### 完成标准

- 内部人员只需记住一套登录体系
- 提交后不会再出现“提示成功但数据不在”的体感
- 登录前后上下文连续

### 为什么必须先做

因为这是整个后台体验最底层的断点。这个问题不收口，后面再怎么改 UI 都会继续让人困惑。

---

## 阶段 2：`submit-stock` 极简化

### 目标

把 `submit-stock` 正式改造成内部 `库存录入台`。

### 本阶段内容

- 删掉或弱化培训式说明
- 删除与当前提交无关的大块摘要区
- 保留极短字段说明
- 明确唯一主动作：`提交并进入 LLM 完善`
- 新增 `market = Global`
- 增加 `market_access_note`
- 梳理字段分组和录入顺序

### 本阶段不做

- 真实 LLM 接入
- 前台详情页结构重做

### 依赖

- 阶段 1 完成

### 完成标准

- 内部人员能快速提交一条记录
- 提交后知道下一步去哪
- 页面首屏信息明显变少，但录入效率更高

### 风险控制

- 不要在这一阶段引入复杂自动补全和复杂联动逻辑
- 表单仍优先服务录入速度

---

## 阶段 3：提报队列与审核页重构

### 目标

把 `/admin/submissions` 和 `/admin/submissions/[id]` 真正收口成：

- `提报队列`
- `LLM 完善与审核页`

### 本阶段内容

#### `/admin/submissions`

- 简化页面说明
- 统一状态文案
- 合并重复含义动作
- 每条记录突出“下一步动作”

#### `/admin/submissions/[id]`

- 顶部明确当前页定位：`LLM 完善与审核`
- 固定处理顺序：补齐 -> AI 建议 -> 审核 -> 转草稿
- 明确展示缺失字段、风险项和下一步去向
- 减少重复信息区块

### 本阶段先不做

- 最终发布动作改造
- 前台内容拆分

### 依赖

- 阶段 2 完成

### 完成标准

- 操作人员从 submit 完成后能清楚进入 review
- 从列表页进入单条处理页后，不再需要猜按钮区别
- review 页第一屏就能告诉人现在要做什么

### 为什么排在这里

因为后台真正的认知阻塞主要发生在“提交之后不知道怎么继续处理”这一段。

---

## 阶段 4：发布前确认页与总控台收口

### 目标

统一 `/admin/edit/[id]` 和 `/admin` 的页面定位，避免总控台继续承载过多主作业动作。

### 本阶段内容

#### `/admin/edit/[id]`

- 明确表达这是 `发布前确认`
- 突出“来自哪条 submission”
- 明确“AI 之后、发布之前”的位置
- 强化发布阻塞项和准备度

#### `/admin`

- 降级为总控入口页
- 收紧重复模块
- 减少“最近录入 / 审核队列 / 最近草稿 / 库存”之间的概念混淆
- 对 `Staff` 强调进入提报队列，对 `Admin` 保留高级入口

### 依赖

- 阶段 3 完成

### 完成标准

- `/admin/edit/[id]` 的定位清楚到不用额外解释
- `/admin` 不再给人“什么都能做，所以什么都不清楚”的感觉

### 当前状态

- 已完成第一轮收口
- 当前进入真实人工 walkthrough 验证期
- 后续只根据实际使用反馈修补，不建议再先做主观加法

---

## 阶段 5：字段分层与前台内容拆分

### 目标

正式解决“中文备注直出前台”“description 混杂”“单位不统一”等结构问题。

### 本阶段内容

- 建立 `原始输入值 / 规则标准化值 / LLM 建议值 / 最终发布值` 的分层心智
- 优先补规则层标准化：
  - `nicotine -> %`
  - `e_liquid -> mL`
  - `puff -> Puffs`
  - brand / market / product_type 的 canonical mapping
- 前台详情页拆为：
  - `Product Overview`
  - `Inventory Manifest`
  - `Commercial Notes`
- 后台同步明确这些内容的来源和去向

### 依赖

- 阶段 4 完成

### 完成标准

- 前台不再直接展示中文原始备注
- 前台单位完整
- manifest 与 notes 分层清楚
- 后台讨论“哪个字段去哪”时有清晰映射

### 为什么不更早做

因为如果前面的页面职责和流程还没稳定，这一层改完仍会因为 review 和 edit 的职责变化而返工。

### 当前状态

- 尚未启动开发
- 保持为后台 walkthrough 稳定后的第一优先后续任务
- 只有在你确认当前后台主链路已经“顺”之后，才建议正式进入

---

## 阶段 6：真实 LLM 第一落点接入

### 目标

让 LLM 开始承担“规则层做不好，但人工又很耗时”的整理工作。

### 第一落点

- `/admin/submissions/[id]`

### 本阶段内容

- 设计结构化 prompt 和结构化 JSON 输出
- 生成：
  - 英文 `Product Overview` 候选
  - 英文 `Commercial Notes` 候选
  - `Inventory Manifest` 整理候选
  - 标准字段候选与风险项
- 人工选择性采用，不自动覆盖最终发布值

### 本阶段不做

- 自动发布
- 在 `submit-stock` 实时调用 LLM
- 前台直接依赖模型即时生成

### 依赖

- 阶段 5 完成
- 冷启动数据已积累到足以验证真实整理效果
- 内部已确认模型供应商与预算边界

### 完成标准

- LLM 明显减少人工整理时间
- 输出稳定可读
- 人工审核负担下降，而不是上升

### 当前备注

- 本阶段当前仅做文档埋点和路线预留
- 在你明确发出“开始接入真实 LLM”指令前，不启动实际接入开发
- 当前仍以前台内容拆分与字段分层完成为前置条件

### 风险控制

- 必须保留人工确认
- 必须可回看原始输入与模型输出差异
- 模型输出异常时不能阻塞整条链路

---

## 阶段 7：全链路内部培训与交接文档

### 目标

在主链路、字段分层和 LLM 第一落点稳定后，再统一整理一份覆盖完整后台流程的内部培训说明。

### 本阶段内容

- 培训文档覆盖 `submit-stock -> admin/submissions -> admin/submissions/[id] -> admin/edit/[id] -> 发布`
- 按真实操作顺序整理，而不是按单页拆散说明
- 记录每一页的主任务、进入方式、退出去向和最小判断标准
- 把页面中删除的教学型说明沉淀到文档，而不是重新塞回 UI
- 补充 `Staff` / `Admin` 的最小职责边界与异常处理方式

### 本阶段不做

- 把培训文档重新拆回页面提示
- 在主链路未稳定前提前编写大而全说明

### 依赖

- 阶段 2 到阶段 6 完成并稳定

### 完成标准

- 新人可以按文档完整走通一次后台主链路
- 页面本身继续保持操作导向，不重新堆积教学文字
- 培训文档与真实页面行为保持一致

### 当前备注

- 本阶段明确放在当前任务收尾后执行
- 当前先继续处理页面与流程，不提前展开培训文档编写
- 当前更适合先把 walkthrough 结果沉淀成简短问题清单，而不是提前写培训手册

---

## 5. 建议的每阶段验收方式

每一阶段结束后，都建议只做一轮轻量人工 walkthrough，不急着补重型自动化。

统一按以下问题验收：

1. 我从哪里进入这一页
2. 我在这一页的唯一主任务是什么
3. 我做完之后系统是否明确告诉我下一步去哪
4. 是否还有同一含义的重复按钮或重复解释
5. 文档描述是否与真实页面一致

## 7. 当前最优先事项

按本文件顺序，最优先的不是“立即大改所有页面”，而是：

1. 统一文档基线
2. 收口登录与提交链路
3. 确保 submit 后能顺畅进入 review

这是整个后台最关键的底盘。

---

## 8. 结论

后台重构的正确顺序不是：

- 先做更多功能
- 先做更多说明
- 先做全量 LLM

而应该是：

- 先把链路拉直
- 再把页面职责切清
- 再把字段分层做对
- 最后让 LLM 接进最合适的一站

后续执行和排期，应统一以本文件为顺序基准。
