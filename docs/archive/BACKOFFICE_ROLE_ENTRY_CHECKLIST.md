## 1. 目的

本文件用于承接后台重构蓝图中的第 4 段：

`最小角色与入口限制`

目标不是引入复杂权限系统，而是在 MVP 范围内把后台入口、默认工作台和关键动作边界收紧到足够清楚、足够安全、足够不易误操作。

---

## 2. 当前现状

基于当前代码实现，后台权限现状如下：

- `/admin` 已支持最小双角色登录：`Admin` / `Staff`
- 登录成功后会写入带签名的后台会话 cookie：`vsh_admin_session`
- 会话中已包含最小角色信息，而不再只是布尔型登录态
- `submit-stock` 已统一纳入后台登录体系，不再单独维护 access code 入口
- `/admin`
- `/admin/submissions`
- `/admin/submissions/[id]`
- `/admin/edit/[id]`
  已能读取并区分 `Admin` / `Staff`

这意味着当前已经落地的最小角色边界包括：

- `Staff` 进入 `/admin` 时看到精简版总控台
- `Staff` 看不到 `手动新建草稿`
- `Staff` 看不到 `高级入口：导入 AI 草稿包`
- `Staff` 在 `/admin/edit/[id]` 只能保存内容，不能切换状态，也不能删除 inventory
- `Admin` 保留完整总控台可见范围、状态切换能力和删除能力

结论：

- 第 4 段“最小角色与入口限制”核心目标已经基本落地
- 当前更需要做的是轻量 workflow review、问题回写和文档同步，而不是继续扩张权限系统

---

## 3. 第 4 段目标态

MVP 只保留两种后台角色：

### 3.1 `Staff`

默认职责：

- 录入提报
- 处理提报队列
- 补齐和标准化 submission
- 生成 AI 建议
- 转成 draft
- 进入草稿页继续修正内容

不应拥有：

- 最终发布型动作
- 高级导入入口
- 日常不需要的后台总控高级能力

### 3.2 `Admin`

默认职责：

- 拥有全部后台可见范围
- 处理最终 draft 质量检查
- 决定是否发布为 `active`
- 使用高级入口
- 处理异常和低频高风险动作

---

## 4. 页面与入口边界

### 4.1 `Staff` 默认应看到

- `/submit-stock`
- `/admin/submissions`
- `/admin/submissions/[id]`
- `/admin/edit/[id]`

### 4.2 `Staff` 默认不应看到或不应拥有

- `/admin` 中的 `手动新建草稿`
- `/admin` 中的 `高级入口：导入 AI 草稿包`
- inventory 状态改成 `active` 的最终发布动作

### 4.3 `Admin` 默认可见

- 全部后台页面
- 全部高级入口
- 全部状态切换动作

---

## 5. 最小实现原则

第 4 段必须遵守以下原则：

- 不引入复杂用户表、邀请机制、账号中心
- 不做细粒度权限矩阵
- 不做多团队、多组织、多供应商后台
- 不把角色系统做成长期认证平台

推荐实现方式：

- 继续沿用当前轻量 cookie 会话
- 在 cookie 中加入最小角色信息
- 先做“页面入口可见性 + 关键动作限制”
- 不急着做复杂中间件总线
- 环境变量最小集合建议为：
  - `BACKOFFICE_SESSION_SECRET`
  - `ADMIN_USERNAME`
  - `ADMIN_PASSWORD`
  - `STAFF_USERNAME`
  - `STAFF_PASSWORD`

## 7. 建议的最小技术方案

在不引入复杂权限系统的前提下，推荐采用：

### 方案 A：双账号 + 单 cookie 角色字段

示意：

- `ADMIN_USERNAME` / `ADMIN_PASSWORD`
- `STAFF_USERNAME` / `STAFF_PASSWORD`
- `BACKOFFICE_SESSION_SECRET`
- 登录成功后写入：
  - 带签名的后台会话 cookie
  - 会话中包含 `role=admin | staff`

优点：

- 最符合当前项目规模
- 代码改动小
- 理解成本低

缺点：

- 仍不是正式用户系统
- 更适合 MVP，而非长期扩展

### 方案 B：单账号 + 环境变量固定角色

示意：

- 只保留一套后台账号
- 通过不同部署环境或手工改环境变量切角色

不推荐原因：

- 不利于同一套环境下区分 `Admin` / `Staff`
- 容易和实际使用场景脱节

结论：

- 默认推荐 `方案 A`

## 8. 第 4 段验收标准

完成后应满足：

- `Staff` 和 `Admin` 的页面心智不同
- `Staff` 默认路径更贴近 `提报 -> 审核 -> AI建议 -> 转草稿`
- `Admin` 默认路径更贴近 `草稿编辑 -> 发布决策`
- `Staff` 看不到或无法执行高级入口与最终发布动作
- 服务端 action 具备最小角色校验，而不只是 UI 隐藏
- 文档、页面、行为三者保持一致
