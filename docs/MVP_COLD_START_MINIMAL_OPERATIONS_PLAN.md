# VapeStockHub MVP 冷启动期极简运营方案

## 1. 文档目的

本方案用于重新拍板当前冷启动阶段的最小运营路径。

目标不是继续把流程拆得更细，而是先用最短路径跑通：

- 供应商库存收集
- 内部快速录入
- LLM 辅助完善
- 人工复核
- 小批量发布上线
- 前台获取有效询盘

一句话目标：

`先上货，再迭代；先人工扛脏活，后考虑自动化。`

---

## 2. 当前正式判断

### 2.1 现在该优先优化什么

当前 MVP 最重要的仍然不是“流程完整度”，而是：

- 尽快把真实库存上线
- 尽快获得真实询盘
- 尽快暴露数据脏点与发布瓶颈

所以当前要优先优化的是：

- 更短的内部操作链路
- 更少的页面往返
- 更清晰的人工拍板点

而不是：

- 更复杂的自动化编排
- 更重的供应商自助系统
- 更细的多角色流程设计

### 2.2 当前正式结论

冷启动阶段正式采用以下判断：

- `Excel` 仍然是外部主入口
- `submit-stock` 不再视为供应商自助产品，而是内部录入台
- LLM 是“整理和补全助手”，不是发布决策者
- 所有记录先进入 `draft`，再由人工确认后发布

---

## 3. 当前正式业务路径

### 3.1 外部入口

冷启动阶段，外部主要资料来源保持简单：

- 供应商 Excel
- 聊天记录
- 零散文本
- 图片或图片来源链接

其中主入口仍然是：

- `Excel`

原因：

- 供应商最容易配合
- 内部最容易统一接收
- 适合冷启动阶段低门槛收集

### 3.2 内部主链路

当前正式采用的内部主链路为：

1. 收到供应商 Excel 或原始资料
2. 内部先做一轮人工整理
3. 录入 `submit-stock`
4. 点击 LLM 按钮生成 `AI Draft Package`
5. 快速复核 AI 输出与关键字段
6. 生成或进入 `draft`
7. 补图、修正文案、确认发布字段
8. 发布到前台

### 3.3 对操作人员的最短心智

对内部操作人员，系统心智统一为：

`打开 submit-stock -> 录入 -> LLM 完善 -> 审核 -> 发布`

说明：

- 这里的“审核”允许落在 `submission review` 与 `draft edit` 两个承接点中
- 这里的“发布”仍然以前台可展示的最终 `inventory active` 为准
- 底层可以保留 `draft` 这一层，但不需要让操作人员感受到流程复杂

---

## 4. `Excel` 与 `submit-stock` 的新定位

### 4.1 `Excel` 的定位

`Excel` 在当前阶段承担：

- 供应商资料收集载体
- 内部整理起点
- LLM 原始输入来源之一

`Excel` 不承担：

- 最终发布系统
- 最终库存数据库
- 最终图片管理

### 4.2 `submit-stock` 的定位

`submit-stock` 在当前阶段承担：

- 内部录入台
- 原始资料结构化入口
- LLM 处理前的统一承接页

`submit-stock` 当前不承担：

- 对外供应商自助产品化
- 最终发布页
- 复杂审批流中心

结论：

- `Excel` 负责把资料拿进来
- `submit-stock` 负责把资料送进系统

### 4.3 真实 LLM 预埋点

为避免冷启动数据跑起来后再重新定义链路，当前先在文档层明确真实 LLM 的后续接入位：

- 第一落点固定为 `/admin/submissions/[id]`
- 主要任务固定为：
  - 英文 `Product Overview` 候选
  - 英文 `Commercial Notes` 候选
  - `Inventory Manifest` 整理候选
  - 缺失字段与风险提示
- 输出形式固定为结构化建议，不直接自动发布

当前阶段先不接真实 LLM，只保留路线和字段/页面预留。

---

## 5. 数据处理原则

### 5.1 当前最小原则

当前不追求一次性把脏数据彻底自动化解决。

正式原则为：

- 脏数据先由内部处理
- LLM 负责加速整理
- 人工保留最终判断

### 5.2 当前最小必填字段

至少应稳定获取：

- `Brand`
- `Model / Product Name`
- `Product Type`
- `Available Qty`
- `Target Market`
- `Warehouse Location`

若以上字段无法确认，则不进入主通道。

### 5.3 当前建议补充字段

建议优先补充：

- `Unit Price (USD)`
- `MOQ`
- `Flavor List`
- `Flavor Breakdown`
- `Stock Notes`
- `Packaging Notes`
- `Extra Notes`
- `Image Links`

---

## 6. LLM 的当前角色

### 6.1 当前正式定位

LLM 当前是：

- 字段整理助手
- 标准化助手
- 草稿生成助手
- 风险提醒助手

LLM 当前不是：

- 自动发布器
- 真实性裁决器
- 价格审批器
- 人工替代者

### 6.2 当前建议让 LLM 做的事

优先让 LLM 负责：

- 字段提取
- 字段映射
- 标题整理
- 描述草稿生成
- 缺失字段提示
- 风险标记
- 人工复核重点提示

### 6.3 当前不交给 LLM 的事

当前不建议交给 LLM：

- 自动决定是否发布
- 自动认定价格是否可信
- 自动认定库存是否真实可卖
- 自动跳过人工直接上线

---

## 7. 图片处理策略

### 7.1 当前正式原则

冷启动阶段，图片按以下方式处理：

1. 外部怎么给都先接住
2. 先不要求收集阶段完成媒体整理
3. 原始图片先作为素材保存
4. 先做文本录入和 LLM 整理
5. 确认值得上架后再处理发布图
6. 最终发布图再上传 Supabase

### 7.2 图片发布红线

正式发布前至少满足：

- 至少 1 张有效图片
- 图片能打开
- 图片和产品主体基本一致
- 图片不是明显无关素材

当前允许：

- 实拍图
- 非棚拍图
- 不完美但可信的图

---

## 8. 人工审核红线

### 8.1 必须人工拍板的字段

以下字段必须人工最终确认：

- `price`
- `quantity`
- `market`
- `warehouse_location`
- 图片有效性
- `contact_visibility`
- 是否发布

### 8.2 发布前阻塞项

以下情况不应直接发布：

- 关键字段缺失
- 图片无效
- 价格异常且未复核
- 商品主体不清
- 描述无法支撑前台展示
- 市场或仓库明显不可信

---

## 9. 与当前后台的关系

### 9.1 当前后台继续保留的部分

当前后台继续保留：

- `submit-stock` 录入能力
- `submission review` 承接能力
- `AI Draft Package` 生成能力
- `draft` 编辑页
- 发布前检查

### 9.2 当前后台要收缩的部分

当前不再把后台描述成复杂链路，而要收缩成一个简单心智：

- 录入
- LLM 完善
- 人工复核
- 发布

### 9.3 当前最值得继续复用的路径

技术上继续复用：

`submit-stock -> submission review -> draft edit -> publish`

运营上统一理解为：

`submit-stock -> LLM -> 审核 -> 发布`

---

## 10. 角色权限策略

### 10.1 是否还需要角色权限

需要，但只保留最小边界。

当前正式建议为：

- 保留 `Admin`
- 保留 `Staff`
- 不再扩展复杂角色体系

### 10.2 为什么不能完全去掉

即使流程已经极简，发布动作仍然具有风险：

- 直接影响前台展示
- 直接影响 SEO 页面质量
- 直接影响询盘质量

因此至少应保留一条最小边界：

- `Staff` 可以录入、整理、生成 LLM 结果、修 draft
- `Admin` 负责最终发布与异常处理

### 10.3 冷启动阶段的现实执行方式

如果当前实际上只有你自己在操作，也可以：

- 日常由同一个人执行全部动作
- 但系统层仍保留最小发布边界

结论：

- 业务上可以一人操作
- 系统上不建议完全去掉发布权限边界

---

## 11. 第一批真实数据推进方式

### 11.1 当前推荐节奏

第一批真实数据继续采用：

- 小批量
- 高质量
- 分波上线

### 11.2 当前推荐顺序

1. 先整理已有 Excel 样本
2. 录入 `submit-stock`
3. 生成 LLM 结果
4. 快速复核并转入 `draft`
5. 补图和修正文案
6. 发布高质量白名单库存
7. 再逐步扩量

---

## 12. 当前正式拍板

当前阶段正式采用以下方向：

1. `Excel` 作为外部主入口
2. `submit-stock` 作为内部录入台
3. 内部先手工处理脏数据
4. LLM 用于加速整理，不替代人工发布
5. 所有库存先进入 `draft`
6. 图片在 draft 阶段后处理并上传
7. 只保留 `Admin / Staff` 最小角色边界
8. 不继续扩张复杂自动化流程

---

## 13. 发给我 Excel 时的固定 Prompt

### 13.1 通用版 Prompt

```text
请处理这个已填写的供应商库存 Excel 模板文件：

文件路径：
[把 Excel 的绝对路径粘贴在这里]

执行目标：
1. 读取并理解整个 Excel
2. 检查每条库存记录的字段完整性
3. 识别 missing fields、risk flags、human review focus
4. 清洗并标准化关键字段
5. 为每条记录整理可送入 LLM 的 Raw Input
6. 基于当前项目约定生成 AI Draft Package 草稿
7. 判断是否适合进入 submit-stock / draft 发布链路

处理要求：
- 遵循 `docs/MVP_COLD_START_MINIMAL_OPERATIONS_PLAN.md`
- 遵循 `docs/DATA_ENTRY_STANDARDS.md`
- 遵循 `docs/FIELD_NORMALIZATION_MAPPING_V1.md`
- 前台字段与生成文案保持英文
- 不臆造品牌、价格、数量、仓库、图片或市场
- `product_type` 按当前拍板统一使用 `Disposable Vape`
- 对不确定内容放入 missing fields、risk flags 或 human review focus
- 价格、数量、市场、仓库、图片有效性、是否发布，仍保留人工终审
```

### 13.2 更省事的短版 Prompt

```text
请按 `docs/MVP_COLD_START_MINIMAL_OPERATIONS_PLAN.md`、`docs/DATA_ENTRY_STANDARDS.md` 和 `docs/FIELD_NORMALIZATION_MAPPING_V1.md` 处理这个 Excel：

[把 Excel 的绝对路径粘贴在这里]

帮我完成：
- 字段完整性检查
- missing fields / risk flags / human review focus 整理
- Raw Input 整理
- AI Draft Package 准备
- 是否适合进入 submit-stock / draft 链路建议
```

`外部尽量简单，内部尽量快；先录入，后 AI；先复核，后发布。`
