# 后台轻量全链路 Walkthrough 清单

> 状态说明：
> - 本文件继续保留，但 walkthrough 前提应以 `BACKOFFICE_WORKFLOW_REFACTOR_V1.md` 为准
> - 推荐在 `BACKOFFICE_REFACTOR_EXECUTION_PLAN.md` 的阶段 3 到阶段 5 收口后再执行新版 walkthrough
> - 后续应按新页面定位和新主链路重写本清单

## 1. 目的

这份清单只用于当前 MVP 阶段的后台主链路复盘。

目标不是做重验收，也不是补复杂自动化，而是快速确认：

- 当前后台主链路能不能顺畅跑通
- `Staff` 和 `Admin` 的职责边界是否清楚
- 页面之间的上下文衔接是否顺手
- LLM 下一步应该优先介入哪个环节

---

## 2. 适用范围

本次 walkthrough 只覆盖以下主链路：

1. `/submit-stock`
2. `/admin/submissions`
3. `/admin/submissions/[id]`
4. `Generate AI Suggestion`
5. `Convert to Draft`
6. `/admin/edit/[id]`

不包含：

- 自动化回归测试
- SEO 页面检查
- 公网用户前台链路
- 复杂异常流程穷举

---

## 3. 建议方式

建议按两条视角各走 1 遍：

### 3.1 `Staff` 视角

目标：

- 确认 `提报 -> 审核 -> AI建议 -> 转草稿 -> 草稿修正` 是否顺手

重点看：

- 是否容易知道下一步做什么
- 是否存在多余跳转和重复操作
- 是否误暴露 `Staff` 不该有的动作

### 3.2 `Admin` 视角

目标：

- 确认 `草稿检查 -> 发布判断` 是否清楚

重点看：

- 是否能快速看懂来源和上下文
- 是否能识别是否适合发布
- 是否保留了必要但不过量的控制能力

---

## 4. 执行前准备

开始前确认：

- 已有可用的 `submit-stock` 访问码
- 已有 `Admin` 和 `Staff` 两套后台凭证
- Supabase 本地或当前联调环境可正常使用
- 至少准备 1 条可用于演练的 submission 数据

建议使用一条“信息不完全但可补齐”的 submission，这样更容易看出当前审核页是否顺手。

---

## 5. 主链路检查清单

### 5.1 `submit-stock`

检查项：

- 是否能顺利进入页面
- 必填字段是否容易理解
- 缺失字段提示是否足够直接
- 提交失败时，草稿是否保留
- 提交成功后，是否能明确知道已进入后台待处理队列

通过标准：

- 录入人不需要猜字段含义
- 失败不会导致已填内容白填
- 提交结果反馈清楚

### 5.2 `/admin/submissions`

检查项：

- 是否能快速看懂当前 submission 队列
- 卡片或列表是否足够支持扫读
- 是否能明确知道哪条应优先处理
- `Review Submission` 是否是最明显主动作

通过标准：

- 运营能在短时间内决定先处理哪条
- 列表不会让人找不到下一步入口

### 5.3 `/admin/submissions/[id]`

检查项：

- 页面是否清楚表达“这是标准化和转换中心”
- 缺失字段提醒是否足够清晰
- 表单是否顺手，长文本输入是否不拥挤
- 右侧辅助区顺序是否稳定
- 原始数据和当前编辑数据是否容易对照

通过标准：

- 审核人员能先补齐最小必填，再决定是否继续 AI
- 页面不会因为说明过多而打断操作节奏

### 5.4 `Generate AI Suggestion`

检查项：

- 按钮是否只在合适时机出现或启用
- AI 输出是否清楚分成固定结构
- 是否能快速看懂 `Candidate Output`
- 是否能快速识别 `Risk Flags`
- 是否能从 `Human Review Focus` 看出人工下一步要检查什么

通过标准：

- AI 输出是“辅助判断”，不是“制造新混乱”
- 人工能基于 AI 更快做决定，而不是被迫重新理解一遍

### 5.5 `Convert to Draft`

检查项：

- 转换动作的前提是否明确
- 转换成功后反馈是否清楚
- 是否能明确跳转或进入后续草稿编辑页
- review 和 draft 之间是否保留上下文连接

通过标准：

- 不会出现“已经转成功，但不知道下一步在哪”的情况
- submission 和 draft 之间关系清楚

### 5.6 `/admin/edit/[id]`

检查项：

- 是否能清楚看到该 draft 来源于哪个 submission
- 是否能继续修正核心字段
- `Staff` 是否只能保存内容，不能执行最终发布或删除
- `Admin` 是否能完成最终状态判断
- 右侧发布准备信息是否足够辅助判断

通过标准：

- `Staff` 和 `Admin` 的页面心智明显不同
- `Admin` 能基于现有信息做最终发布判断

---

## 6. 角色边界专项检查

### 6.1 `Staff`

确认以下内容：

- 登录后看到的是精简版 `/admin`
- 看不到 `手动新建草稿`
- 看不到 `高级入口：导入 AI 草稿包`
- 在 `/admin/edit/[id]` 不能切换到 `active`
- 在 `/admin/edit/[id]` 不能删除 inventory

### 6.2 `Admin`

确认以下内容：

- 登录后看到完整总控台
- 可以看到高级入口
- 可以做状态切换
- 可以删除 inventory

---

## 7. 记录方式

每走完一条链路，只记录 3 类结果：

- `顺`：可以连续完成，不需要解释
- `卡`：能做，但需要停下来想一下
- `断`：无法继续，或上下文明显丢失

建议不要写成长篇报告，只记这些最小信息：

- 页面
- 问题点
- 属于 `顺 / 卡 / 断`
- 是否影响业务跑通

---

## 8. Walkthrough 结束后的判断

如果结果大部分是 `顺`，说明当前后台主链路已经具备继续接入 LLM 的前提。

如果结果大量是 `卡`，说明应先继续优化页面结构、提示和上下文衔接。

如果结果出现 `断`，说明应优先修链路，不要提前推进 LLM。

---

## 9. LLM 介入判断口径

walkthrough 完成后，用下面的顺序决定 LLM 第一落点：

### 9.1 优先介入 `/admin/submissions/[id]`

适用前提：

- 当前最大耗时在字段整理、风险归纳、候选草稿提炼
- 人工 review 节奏已经稳定
- 只是希望 AI 帮忙提炼，而不是替代判断

适合让 LLM 先做：

- 结构化字段补全建议
- 风险归纳
- 英文 listing 候选输出

### 9.2 其次介入 `/admin/edit/[id]`

适用前提：

- draft 来源和上下文桥接已经足够清楚
- 人工主要耗时在润色标题、描述和 listing 文案

适合让 LLM 先做：

- title 候选
- description 候选
- 语义整理和英文润色

### 9.3 暂不建议交给 LLM

- 自动决定是否发布
- 自动替代人工做最终 source 可信度判断
- 自动决定价格真实性

---

## 10. 当前建议结论

当前最合理的顺序是：

1. 先按本清单跑一轮轻量 walkthrough
2. 只记录 `顺 / 卡 / 断`
3. 先修真正影响业务跑通的断点
4. 再确定 LLM 第一介入点

这样成本最低，也最符合当前 MVP 阶段目标。
