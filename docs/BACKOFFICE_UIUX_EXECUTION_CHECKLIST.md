## 1. 目的

本清单用于承接后台蓝图中的第 3 段：

`后台中文 UI/UX 重构`

当前目标不是继续扩张后台，而是用最小改动让现有页面更像清晰的工作台，更适合日常运营连续处理任务。

---

## 2. 当前判断

结合现有页面实现，当前后台已经具备以下基础：

- `/admin` 已具备总控台角色分流心智
- `/admin/submissions` 已具备提报队列的最小筛选与进入审核能力
- `/admin/submissions/[id]` 已具备审核、AI 建议、转草稿的主链路
- `/admin/edit/[id]` 已具备草稿编辑和发布限制的最小边界

因此，接下来的 UI/UX 工作应遵守两条原则：

- 只做能提升日常处理效率的改动
- 不为了“看起来完整”而额外增加复杂交互

---

## 3. 执行约束

继续推进时，统一遵守以下约束：

1. 所有后台标题、说明和操作提示继续保持中文短句
2. 每个页面只保留一个最强主动作
3. 右侧信息区尽量固定成相似结构，避免每页认知重新切换
4. 原始数据、AI JSON、长说明继续默认折叠
5. 不为了验收而引入重型自动化或复杂交互壳

---

## 4. 按业务优先级排序

### P0：只做直接影响业务跑通的项

#### 4.1 统一每页“最强主动作”

目标：

- 避免一个页面同时出现多个视觉强度相近的按钮
- 让运营人员一眼知道“当前下一步应该点哪里”

建议动作：

- `/admin`：继续强调默认入口，不再让低频动作抢主视觉
- `/admin/submissions`：主动作保持 `进入审核`
- `/admin/submissions/[id]`：主动作根据状态聚焦到 `保存审核` 或 `转为库存草稿`
- `/admin/edit/[id]`：主动作保持 `保存修改`，发布属于状态动作，不额外做第二主按钮

完成标准：

- 每页首屏只有一个最强 CTA
- 次要动作全部降级为边框按钮、文字链接或折叠项

当前状态：

- 已基本完成
- `/admin/submissions` 主动作已聚焦到 `进入审核`
- `/admin/submissions/[id]` 主路径已围绕 `保存审核` 与 `转为库存草稿`
- `/admin/edit/[id]` 主动作维持 `保存修改`，状态切换保留为受角色限制的状态动作

#### 4.2 收紧长说明和提示密度

目标：

- 减少首屏解释负担
- 保留必要提示，但不让页面重新变成操作手册

建议动作：

- `/admin`：保留角色说明，但压缩重复解释
- `/admin/submissions`：保留“队列说明”，弱化重复引导文案
- `/admin/submissions/[id]`：将重复的审核说明收敛到固定提示面板
- `/admin/edit/[id]`：将角色限制和发布说明集中在顶部与右侧，不重复散落

完成标准：

- 同一意思不在一个页面重复出现 2 次以上
- 首屏说明控制在 2 到 3 句短句

当前状态：

- 已在 `/admin/submissions/[id]` 完成第一轮收紧
- 审核页说明密度已下降，避免重新变成长说明操作手册
- 后续只需按实际使用反馈继续微调，不建议再大改

#### 4.3 固定右侧辅助区结构

目标：

- 降低页面切换后的认知成本
- 让审核页和编辑页的右侧都承担“辅助判断”，而不是临时堆内容

建议动作：

- 审核页右侧固定为：审核状态、缺失字段、AI 建议、转草稿条件
- 编辑页右侧固定为：发布准备度、质量检查、关联来源、操作提醒

完成标准：

- 右侧信息区具备稳定顺序
- 二级信息默认折叠，避免右栏过长

当前状态：

- 审核页右侧结构已基本固定为：审核状态、缺失字段、AI Assist、原始快照
- 草稿页右侧已补齐发布准备度、来源 submission 与 AI 风险桥接
- 当前主要任务不再是继续加模块，而是确认这套结构在日常流转里是否顺手

---

### P1：提升连续处理效率的项

#### 4.4 优化列表卡片的信息层级

目标：

- 让提报队列和库存列表更适合批量扫读

建议动作：

- 列表第一行只放最有判断价值的信息：品牌、型号、状态
- 第二行承接供应商、市场、仓库、数量
- 把“待补字段数”“是否已转草稿”做成更稳定的小状态块

完成标准：

- 同一列表中的主要字段顺序固定
- 扫一眼能判断“是否可继续处理”

当前状态：

- 已完成第一轮
- `/admin` 与 `/admin/submissions` 已补上“处理优先级”与“下一步动作”提示
- 当前应先通过真实操作确认这套层级是否顺手，再决定是否继续微调

#### 4.5 统一状态文案和色彩语义

目标：

- 避免页面之间同一状态看起来不像同一件事

建议动作：

- `new / reviewing / ready / converted`
- `draft / active / reserved / sold`
- 缺失项、阻塞项、可继续项的颜色表达统一

完成标准：

- 相同状态在不同页面使用同类色彩和同类措辞

当前状态：

- 已完成第一轮
- 主流程状态文案、成功提示、错误提示已统一到中文运营口径
- 后续仅根据 walkthrough 反馈修正局部不一致项

#### 4.6 补齐从一个页面跳到下一页面的“上下文提醒”

目标：

- 让运营人员知道自己为什么来到当前页，以及做完后该去哪

建议动作：

- 审核页顶部明确“来自提报队列”
- 草稿页顶部明确“来自哪个 submission”
- 已转草稿后，审核页和草稿页之间互相保留清晰入口

完成标准：

- 关键页面都能回答“我从哪里来，下一步去哪里”

当前状态：

- 已完成第一轮
- 主链路页面已补来源返回、首屏下一步提示和跨页衔接文案
- 当前重点转为验证真实流转是否仍有 `卡` 点，而不是继续加提示

### P2：体验优化项

#### 4.7 统一表单区块的标题与节奏

目标：

- 提升长表单阅读节奏

建议动作：

- 表单区块统一采用：区块标题 + 一句说明 + 字段组
- 避免有的区块说明很长，有的区块完全没有说明

#### 4.8 减少视觉噪音

目标：

- 保持 Dark Pro 的干净、数据密集风格

建议动作：

- 减少过多相似边框盒子的嵌套
- 避免同屏同时出现太多强调色
- 低优先级内容尽量用更弱的文字层级表达

---

## 5. 不建议当前投入的项

以下内容当前不建议继续投入：

- 为后台补复杂端到端自动化验收
- 为角色系统增加更多角色或权限矩阵
- 新增独立 workbench 路由体系
- 大规模重写现有页面结构
- 为后台做“培训式”长文案说明区

---

## 6. 建议的实际推进顺序

接下来更适合按以下顺序逐项收口：

1. 先做后台主链路的轻量 walkthrough，确认 `submit-stock -> queue -> review -> draft -> edit` 是否顺手
2. 再只修 walkthrough 中暴露出的 `卡 / 断` 问题
3. 然后再决定是否进入字段分层与前台内容拆分
4. 最后才考虑真实 LLM 第一落点接入

原因：

- 审核页和草稿页最接近日常主链路
- 这两页的优化，对“业务能否顺畅跑通”影响最大
- 当前页面层收口已基本完成，下一步价值主要来自真实使用反馈

---

## 7. 当前推荐下一小步

如果按“最小投入、最大收益”原则继续，下一小步建议是：

`先跑一轮轻量全链路 walkthrough，再决定是否继续进入字段分层和真实 LLM`

这是当前最值钱的一步，因为它直接影响：

- 现有后台工作流是否真的顺手
- 文档、页面、行为是否一致
- 字段分层是否已经具备开工条件
- LLM 应该先辅助 review，还是继续保持路线预留
- 是否还有阻塞业务跑通的隐性断点
