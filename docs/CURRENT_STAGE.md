# VapeStockHub 当前阶段与功能归类规则

更新时间：2026-05-21

## 1. 文档目的
- 本文用于给当前项目提供一份统一、面向执行的阶段说明。
- 目标不是重复历史规划，而是回答以下三个问题：
  - 当前项目到底处于什么阶段
  - `MVP` 什么时候算正式结案
  - 新需求应归类为 `MVP 收尾`、`MVP+ 优化` 还是 `第二阶段扩张`
- 后续任何 Agent、协作者或新会话进入项目时，应优先阅读本文，再继续判断需求优先级。

## 2. 当前阶段结论
- 当前项目状态不是“还在做最初的未上线 MVP”。
- 当前项目状态应统一定义为：
  - `MVP 主体已完成`
  - `正式站点已上线`
  - `当前处于 MVP 收尾 + 上线后早期优化阶段`
- 更简洁的对外执行口径建议统一为：
  - `当前阶段：MVP 收尾阶段`

## 3. 为什么这样定义
- 仓库已有审计与收口文档明确说明：
  - `MVP` 核心功能已完成
  - 站点已具备真实可访问形态
  - 当前主要缺口不在“主体功能没做完”，而在“收尾、质量保障、运营闭环、文档一致性”
- 因此，后续讨论功能时，不应继续沿用“上线前 6 周执行计划”的旧语境。
- `docs/MVP_PLAN.md` 现在更适合作为历史启动基线。
- 当前更应该以以下文档共同作为主判断依据：
  - `docs/MVP_AUDIT_REPORT.md`
  - `docs/MVP_CLOSURE_CHECKLIST.md`
  - `docs/PARKING_LOT.md`

## 4. 当前阶段的主目标
- 当前阶段的主目标不是继续堆叠中长期功能。
- 当前阶段的主目标是把项目从“已经上线、主体可用”推进到“可稳定运营、可稳定承接真实询盘、可阶段性结案”。
- 这一阶段优先关注：
  - 核心链路稳定性
  - 最小质量门禁
  - 运营复核节奏
  - 文档与现实状态一致
  - 上线后高价值优化项

## 5. `MVP` 正式结案判断
- `MVP` 正式结案，不以“页面已经上线”作为唯一标准。
- 对本项目，`MVP` 更合理的结案条件是：
  1. `P0` 收尾项全部完成
  2. `P1` 质量保障项完成
  3. `P1` 运营闭环项已开始执行
  4. 真实站点已跑过至少一轮稳定观察周期
  5. 当前未收尾事项已变成“明确完成”或“明确延期”，而不是模糊状态
- 在满足以上条件前，项目可表述为：
  - `MVP 已基本做完`
  - `MVP 主体完工`
  - `MVP 收尾阶段`
- 在满足以上条件后，才建议表述为：
  - `MVP 阶段正式结案`

## 6. 需求归类规则

### 6.1 `MVP 收尾`
- 定义：不做会影响当前 `MVP` 阶段结案或影响线上主链路稳定性。
- 判断标准：
  - 是否直接影响公开站核心链路可用性
  - 是否直接影响后台发布链路可用性
  - 是否属于监控、部署、质量门禁、复核机制、文档对齐
  - 是否会影响真实询盘承接与基本运营
- 典型例子：
  - 正式环境 walkthrough
  - 监控告警接通
  - 生产环境变量核对
  - 最小 CI / smoke test
  - 真实库存复核节奏
  - 有效询盘口径统一

### 6.2 `MVP+ 优化`
- 定义：不影响 `MVP` 结案，但可以提升现有主链路的发现效率、转化效率、运营效率或数据质量。
- 判断标准：
  - 不改变当前商业模型
  - 不引入新的复杂角色体系
  - 不引入明显超出当前组织能力的新流程
  - 仍然服务于北极星指标：`有效询盘数`
- 典型例子：
  - 轻量库存搜索
  - 结果排序优化
  - 列表页筛选增强
  - CTA 文案与位置优化
  - 后台工作台检索与操作提效
  - 高价值页面展示细节优化

### 6.3 `第二阶段扩张`
- 定义：开始新增业务能力、供给入口、角色复杂度或平台化能力，已经超出当前 `MVP` 收尾和早期优化范围。
- 判断标准：
  - 引入新的业务角色或复杂权限
  - 引入新的供给侧机制
  - 引入新的平台化流程
  - 引入超出当前团队承接能力的系统复杂度
  - 目标从“提升询盘”转向“扩张平台能力”
- 典型例子：
  - 开放卖家注册与入驻
  - 复杂用户管理和角色矩阵
  - 自动化 SLA 系统
  - 多语言大规模扩张
  - 商业化收费系统
  - 在线交易、支付、订单闭环

## 7. 快速判断问题清单
- 每次有新需求，先问以下问题：
  1. 这个功能不做，会不会影响当前 `MVP` 结案？
  2. 这个功能做了，是否仍然围绕“库存展示 -> 私域联系 -> 有效询盘”主链路？
  3. 这个功能会不会引入新的角色、系统复杂度或平台化能力？
  4. 这个功能是解决“当前已上线产品的真实问题”，还是提前建设中长期能力？
  5. 当前团队是否能用最小复杂度把它做完并维护？
- 判断口径：
  - 第 1 题答“是”：优先归入 `MVP 收尾`
  - 第 2 题答“是”，且第 3 题答“否”：优先归入 `MVP+ 优化`
  - 第 3 题答“是”或第 4 题明显偏中长期：优先归入 `第二阶段扩张`

## 8. 搜索功能示例判断
- 以当前项目状态看，搜索功能不属于“缺了就无法结案”的 `MVP 收尾` 项。
- 但搜索功能也不天然属于“超出边界的第二阶段扩张”。
- 如果搜索的范围控制在以下边界内，应归类为 `MVP+ 优化`：
  - 只服务于现有库存发现和后台提效
  - 不做全站复杂搜索
  - 不引入重型搜索基础设施
  - 不新建一套复杂 SEO 搜索结果体系
- 当前更推荐的搜索范围：
  - 公开站：`/inventory` 轻量关键词搜索
  - 后台：`/admin` 与 `/admin/submissions` 工作台搜索
- 当前不推荐的搜索范围：
  - 全站统一搜索
  - 博客/政策/库存混合搜索
  - 联想推荐、复杂相关度排序、重型搜索引擎接入

## 9. Agent 协作规则
- 后续任何 Agent 接手需求时，应先默认当前项目处于 `MVP 收尾阶段`。
- 若没有新的拍板记录，不应把项目误判为“最初未上线 MVP 阶段”。
- 若需求存在争议，按以下顺序判断：
  1. 先看本文
  2. 再看 `docs/MVP_AUDIT_REPORT.md`
  3. 再看 `docs/MVP_CLOSURE_CHECKLIST.md`
  4. 再看 `docs/PARKING_LOT.md`
- 若需求明确超出本文定义的 `MVP 收尾` 和 `MVP+ 优化`，应先记录到 `docs/PARKING_LOT.md` 或单独形成第二阶段讨论文档，再决定是否实施。

## 10. 当前执行建议
- 当前优先级建议按以下顺序理解：
  1. 先完成剩余 `MVP 收尾`
  2. 再做高价值 `MVP+ 优化`
  3. 再进入第二阶段扩张议题
- 若新增需求能明显提升真实询盘效率，但不会引入额外系统复杂度，可在收尾阶段并行纳入 `MVP+ 优化` 评估。

## 11. 一句话口径
- `VapeStockHub 当前不是“未完成的 MVP 项目”，而是“已上线、MVP 主体完成、正处于收尾与早期优化阶段”的项目。`
